# Handoff

세션을 durable 지식 노드로 **checkpoint**하는 규율이다. 목적은 하나 — *다음 세션(또는 다른 사람)이 컨텍스트 없이도 즉시 이어받게 한다*.

비유: 코드 커밋이 작업 산출물을 저장한다면, 핸드오프는 *작업 중인 머릿속 상태*를 저장한다.

핸드오프는 `okf-authoring` 스킬의 규율을 세션 상태에 적용한 것이다 — 복제가 아니라 *이어받는 데 필요한 맥락만*.

---

## 1. 3층 모델 (섞지 않는다)

성격이 다른 세 층을 분리하는 게 핵심이다. 섞으면 다음 세션이 "지금 뭘 해야 하는지"를 못 찾는다.

| 층 | 성격 | 위치 | 갱신 |
|---|---|---|---|
| **완료 작업** | 영구·append-only | `docs/history/` (`type: History`) | 작업 단위마다 새 노드 추가 |
| **진행+남은 작업** | 살아있는 스냅샷 | `docs/knowledge/session-state.md` (`type: Handoff`) | 매 핸드오프마다 **덮어씀** |
| **외부 트래커** | 프로젝트별 SSOT | issue·notion·linear·file | 상태 전이 + 상태 코멘트 |

원칙: **완료는 아카이브(History), 진행은 스냅샷(resume), 진짜 작업목록은 트래커.** resume 노드는 "지금 상태 사진"이지 작업 대장이 아니다.

---

## 2. resume 노드 포맷 (`type: Handoff`)

한 프로젝트에 하나. 매 핸드오프마다 덮어쓴다. 맨 위 "여기서 시작"이 가장 중요하다.

```markdown
---
type: Handoff
title: <프로젝트> 세션 상태
description: 진행 중·남은 작업 스냅샷 (매 핸드오프 갱신)
timestamp: <ISO8601, KST +09:00>   # 실제 시각 조회해 기록
resource: <branch> @ <git SHA 7자리>
tags: [handoff, session-state]
status: in-progress | blocked | ready-to-resume
---

# <프로젝트> 세션 상태

## ▶ 여기서 시작
<다음 단일 액션 한 줄 — 다음 세션이 제일 먼저 읽는 곳>

## 진행 중
- <현재 작업 + 어디까지 했는지>

## 남은 작업
- [ ] <할 일> — <왜/맥락>

## 블로커 · 열린 루프
- <기다리는 것(CI·리뷰·외부 의존) + 어디서 확인하는지>

## 이번 세션 결정 (근거)
- <결정>: <이유> — 다음 세션이 재논의하지 않게

## 함정 · 주의
- <발견한 트랩> — future-you 시간 절약 (예: "릴리즈는 develop+태그, main은 방치")

## 부작용 (이 세션이 만든 것)
- 브랜치/PR/태그/배포/외부 전송: <고아 방지용 목록>

## 검증 상태
- build / test / deploy: <실제 결과>

## 완료 (History 링크)
- [<작업>](../history/<파일>.md)
```

## 3. 무엇을 담나 / 안 담나

### 담는다 (이어받는 데 필요한 것)
- **다음 액션** — 단일 문장. 이거 하나로 10초 안에 재개
- **진행 중 상태** — 어디까지 했고 다음 스텝은 무엇인지
- **블로커·열린 루프** — 기다리는 것과 확인 위치
- **결정+근거** — 재논의 방지
- **함정** — 이 세션이 발견한 트랩
- **부작용** — 만든 브랜치/PR/태그/배포 (고아 리소스 방지)
- **검증 상태** — build/test/deploy 결과 (재작업·잘못된 가정 방지)

### 안 담는다
- **대화 전체 복붙** — 요약이지 트랜스크립트가 아니다
- **코드 복제** — 상세는 파일 경로·SHA로 가리킨다 (`okf-authoring` 스킬 싱크 회피 원칙 계승)
- **완료 작업의 장문 서술** — 그건 History 노드로. resume엔 링크만
- **개인·비공개 정보** — resume 노드는 커밋·공유 대상 (아래 4)

---

## 4. 개인 메모리와의 구분 (선택적 레이어)

- **resume 노드(OKF Handoff)** = 프로젝트·**커밋·팀 공유**·SSOT. 자립적으로 동작한다.
- **개인 메모리 도구**(remember 등) = 개인·비공개·크로스프로젝트·휘발.

둘은 **의존이 아니라 레이어**다. 개인 메모리 도구가 있으면 handoff가 그 버퍼에 *포인터 한 줄*만 얹어 SessionStart 자동 로드에 태울 수 있다(선택). 없어도 resume 노드만으로 정상 동작한다. **하드 의존을 만들지 않는다** — 배포·이식성·관심사 분리를 해친다.

경계: 자격증명·개인정보·한 세션에만 의미 있는 것은 resume 노드(커밋 대상)에 넣지 않는다.

---

## 5. 태스크 매니저 비종속

작업 목록은 프로젝트마다 다른 도구(issue·notion·linear·file)로 관리된다. handoff는 **어댑터**로 대한다.

- **감지**: `.okf/handoff.yml` 설정이 있으면 그대로, 없으면 자동 추정(`gh` 있으면 github-issues, Notion MCP 연결 시 notion, `docs/tasks.md` 있으면 file, 아무것도 없으면 none).
- **동작**: 진행 중 항목 상태 전이 + 상태 코멘트. **외부로 나가는 코멘트는 draft를 보여주고 확인**받는다(설정으로 auto 가능).
- `none`이면 외부 동기화를 건너뛰고 resume 노드만 남긴다.

---

## 6. 자기 점검 (마감 전)

- 다음 세션이 **"여기서 시작" 한 줄로 즉시 재개**할 수 있는가
- 완료 작업마다 History 노드가 있는가 (없으면 초안·커밋)
- 만든 브랜치/PR/태그/배포가 **부작용 목록**에 다 있는가 (고아 없음)
- 검증 상태(build/test/deploy 결과)가 사실대로 적혔는가 — "됐다"고 얼버무리지 않는다
- 대화 복붙·코드 복제 없이 *맥락*만 담았는가
- resume 노드에 커밋하면 안 되는 개인·비밀 정보가 없는가

---

## 7. 언제 스킵하나

넘길 상태가 없는 세션(단순 질의·사소한 수정)은 ceremony를 생략한다. 핸드오프는 *이어받을 게 있을 때*의 도구다. 애매하면 resume 노드의 "여기서 시작"만 갱신하고 나머지는 생략한다.
